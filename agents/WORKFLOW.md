# 内容工厂 Agent 协作流程

> 8 层角色，1 条「战略 → 设定 → 剧本 → 生产 → 审稿 → 运营 → 商务 → 复盘」闭环 DAG。
> 支持：条件分支、质疑协议、共享记忆、DAG 编排、反馈闭环。
> **v2 更新：级联协议 + 反馈闭环协议——关键交接点自动走查，防止方向跑偏。**

---

## 核心闭环 DAG

不是线性流水线，而是支持条件分支的有向无环图。审稿是关键闸门，复盘驱动下一轮：

```
@主编 定义用户画像 + 情绪洞察 + 选题决策
  │
  └── ⚡ 董事长走查选题 ← 反馈闭环①（≤2轮）
        │
        ├── 通过 → @设定师 构建世界观 + 人物 + 关系 + 风格
        │           │
        │           └── ⚡ 编剧走查设定 ← 反馈闭环④（≤2轮）
        │                 │
        │                 ├── 通过 → @编剧 设计故事弧线 + 长线规划 + 爆点 + 连载节奏
        │                 │           │
        │                 │           └── ⚡ 写手走查剧本 ← 反馈闭环⑤（≤2轮）
        │                 │                 │
        │                 │                 ├── 通过 → @写手 场景设计 + 对话写作 + 章节生成 + 文风统一
        │                 │                 │           │
        │                 │                 │           └── @审稿 结构审查 + 情绪强化 + 节奏优化 + 一致性检查 ← 反馈闭环②
        │                 │                 │                 ├── 审稿通过(publish_ready=true) → @运营
        │                 │                 │                 └── 审稿不达标(P0/P1) → @写手 修改重写（≤2轮）
        │                 │                 │
        │                 │                 └── 打回 → @编剧 修改剧本
        │                 │                       └── 2轮打回 → 上报董事长
        │                 │
        │                 └── 打回 → @设定师 修改设定
        │                       └── 2轮打回 → 上报主编
        │
        └── 打回 → @主编 修改选题
              └── 2轮打回 → 上报董事长

@运营 读者转化 + 社群设计 + 留存 + 传播
      │
      └── @商务 会员设计 + 付费墙 + 产品化 + IP运营
            │
            ├── 💰 付费墙位置人工确认 ← 人工决策点
            │
            └── @复盘官 反馈收集 + 内容复盘 + 技能优化 + 经验沉淀
                  │
                  └── 反馈闭环③ → @主编（新一轮内容生产）
                        ├── 继续 → 下一轮优化
                        ├── 调整 → 修改选题/设定
                        └── 砍掉 → @主编 切品类 → 回到 L0 重新定义
```

### 质疑节点（按 challenge-protocol.md）

```
@主编 出选题决策 ──→ @设定师 质疑设定可行性 ──→ 通过/修改/打回
                 ──→ @编剧 质疑故事可展开性 ──→ 通过/修改/打回

@设定师 出设定 ──→ @编剧 质疑设定是否足以支撑 50+ 章故事 ──→ 通过/修改/打回

@编剧 出剧本 ──→ @写手 质疑场景可执行性 ──→ 通过/修改/打回
              ──→ @审稿 质疑节奏和爆点是否合理 ──→ 通过/修改/打回

@写手 出章节 ──→ @审稿 质疑一致性和情绪强度 ──→ 通过/修改/打回

@商务 出付费墙位置 ──→ @运营 质疑是否打断读者体验 ──→ 通过/修改/打回
```

## 角色职责速查

| 角色 | 调用 | 做什么 | 不做什么 |
|------|------|--------|----------|
| **主编** | `@主编` | 定方向、画像、选题、情绪 | 不陷入写作细节 |
| **设定师** | `@设定师` | 世界观、人物、关系、风格 | 不自己写故事线 |
| **编剧** | `@编剧` | 故事弧线、伏笔、爆点、节奏 | 不亲自写每一章 |
| **写手** | `@写手` | 场景、对话、章节、文风 | 不改设定不改故事线 |
| **审稿** | `@审稿` | 结构、情绪、节奏、一致性 | 不跳过审查直接发布 |
| **运营** | `@运营` | 转化、社群、留存、传播 | 不刷量不追虚荣指标 |
| **商务** | `@商务` | 会员、付费墙、产品化、IP运营 | 不在平淡处卡付费墙 |
| **复盘官** | `@复盘官` | 反馈、复盘、优化、沉淀 | 不写流水账，要提取规则 |

## 快速出作品流程（1 周）

> 内容工厂铁律：没有可发布的章节，就没有读者。

```
Day 1: @主编 定画像+情绪+选题 + @设定师 出世界观+人物
       输入：目标平台 + 品类倾向
       输出：PersonaSheet + EmotionMap + TopicDecision + WorldBook + CharacterCard[]

Day 2: @设定师 出关系+风格 + @编剧 出故事弧线+爆点
       输入：L0+L1 产出
       输出：RelationMap + StyleGuide + StoryArc + HighlightList + RhythmChart

Day 3-4: @写手 批量产章节（5-10 章）
       输入：StoryArc + StyleGuide + RhythmChart
       输出：FinalChapter[]（5-10 章）

Day 5: @审稿 全链路审查 + @运营 出转化+传播方案
       输入：FinalChapter[] + 上游所有资产
       输出：publish_ready = true + ConversionPlan + SpreadPlan

Day 6-7: @商务 出付费墙+会员 + 发布上线 + 开始收集反馈
       输出：PaywallStrategy + SubscriptionPlan + 读者数据流
```

### 交接标准

| 交接 | 从 → 到 | 交接物 | 验收条件 |
|------|---------|--------|----------|
| 主编 → 设定师 | 选题 → 设定 | PersonaSheet + EmotionMap + TopicDecision | 画像有 3 个可验证假设，选题有差异化论证 |
| 设定师 → 编剧 | 设定 → 剧本 | WorldBook + CharacterCard[] + RelationMap + StyleGuide | 世界观有 3 条规则，角色有动机和缺陷 |
| 编剧 → 写手 | 剧本 → 生产 | StoryArc + LongTermPlan + HighlightList + RhythmChart | 故事有三幕结构，爆点间距 ≤ 5 章 |
| 写手 → 审稿 | 生产 → 审校 | FinalChapter[] | 每章 3000-5000 字，开头 200 字有钩子 |
| 审稿 → 运营 | 审校 → 发布 | publish_ready = true | P0/P1 问题清零，一致性 100% |
| 运营 → 商务 | 发布 → 变现 | ConversionPlan + CommunityPlan + SpreadPlan | 转化漏斗有量化目标 |
| 商务 → 复盘官 | 变现 → 复盘 | SubscriptionPlan + PaywallStrategy | 付费墙卡在情绪高点 |
| 复盘官 → 主编 | 复盘 → 新轮 | FeedbackDB + ReviewBook + KnowledgeBase | 每轮 ≥ 3 条可执行规则 |

## 一周一迭代节奏

```
周一（30min）          周三（15min）          周五（45min）
┌──────────┐         ┌──────────┐         ┌──────────────┐
│ 本期选题   │────────→│ 生产检查  │────────→│ 审稿+复盘    │
│ @主编 主导 │        │ @写手 反馈 │        │@审稿+@复盘官  │
│ @设定师 确认│       │ @编剧 协调 │        │@主编 校准    │
│ @编剧 可行  │       │           │        │              │
└──────────┘         └──────────┘         └──────────────┘
```

### 周一规划

- @主编 从 OKR 提取本期选题方向
- @设定师 确认设定就绪
- @编剧 确认故事弧线 + 本期爆点
- @写手 确认章节产出计划
- 质疑：@编剧 对选题做故事可展开性评审

### 周三检查

- @写手 反馈生产障碍（设定不够细/角色动机不清/节奏对不上）
- @编剧 协调调整
- 偏航 → 当天调整，不等到周五

### 周五复盘

- @审稿 出审稿报告（结构/情绪/节奏/一致性）
- @主编 判断内容假设是否成立
- @复盘官 提取本轮经验教训
- @主编 做战略校准（如需要）
- 定下期方向
- 写入 `.claude/blackboard/current-sprint.md`

## 紧急流程

### 严重一致性事故（人物/世界观前后矛盾已发布）

```
@审稿/读者 发现事故 → @写手 定位矛盾章节 → @审稿 评估影响范围
                     → @写手 修复 → @审稿 验证 → 重新发布
全程 < 2 小时止血
```

### 内容方向错误（读者大面积流失/差评）

```
@复盘官 报告数据异常 → @主编 判断是内容问题还是运营问题
                     → 内容问题 → @编剧 调整故事走向 + @写手 修改
                     → 方向问题 → @主编 切品类 → 回到 L0
不靠"再写几章看看"硬撑，数据说话
```

### 爆款机会窗口（某内容意外爆火/平台推流）

```
发现爆火信号 → @运营 加大传播投入 → @商务 提前设计付费墙和产品化
             → @主编 评估是否衍生新 IP → @设定师 快速扩展设定
抓住窗口期，48 小时内出商业方案
```

## 级联协议（Cascade Protocol）

Agent 完成核心工作后，自动检测下游是否存在，自动派发下游 Agent。用户只需和入口角色（@主编）沟通，整条链自动走完。

### 级联路由表

| 当前 Agent | 完成条件 | 下游 Agent | 交接物 | 条件 |
|-----------|---------|-----------|--------|------|
| @主编 | 选题决策+画像+情绪写入 blackboard | @设定师 | PersonaSheet + EmotionMap + TopicDecision | 交付型任务，董事长走查通过 |
| @设定师 | 世界观+人物+关系+风格写入 blackboard | @编剧 | WorldBook + CharacterCard[] + RelationMap | 设定完成，编剧走查通过 |
| @编剧 | 故事弧线+长线规划写入 blackboard | @写手 | StoryArc + LongTermPlan + HighlightList | 剧本完成，写手走查通过 |
| @写手 | 章节定稿写入 blackboard | @审稿 | FinalChapter[] | 写作完成 |
| @审稿 | publish_ready = true | @运营 | 审核通过的章节 | 审稿通过 |
| @审稿 | publish_ready = false | @写手 | 修改要求 + P0/P1 问题 | 审稿打回（≤2 轮） |
| @运营 | 转化+社群+留存方案写入 blackboard | @商务 | ConversionPlan + CommunityPlan | 运营方案完成 |
| @商务 | 付费+产品化方案写入 blackboard | @复盘官 | SubscriptionPlan + PaywallStrategy | 商务方案完成 |
| @复盘官 | 反馈+复盘+经验写入 blackboard | @主编 | FeedbackDB + ReviewBook + KnowledgeBase | 复盘完成（闭环） |

### 级联触发判断

| 任务意图 | 级联？ |
|---------|--------|
| 来自上游 Agent 的级联任务 | ✅ 级联 |
| 包含"走完流程""全流程""从选题到发布""一键出内容"意图 | ✅ 级联 |
| 单一动作（"写个章节""做个审稿"） | ❌ 不级联 |
| 用户说"只做这一步" | ❌ 不级联 |

### 级联-闭环交互

```
交付型任务 → @主编 → 董事长走查选题 ──→ 通过 → @设定师
                                    │           │
                                    │           └── 编剧走查设定 ──→ 通过 → @编剧
                                    │                                     │
                                    │                                     └── 写手走查剧本 ──→ 通过 → @写手 → @审稿
                                    │                                                        │
                                    │                                           ┌──────────┘
                                    │                                           │
                                    └── 打回 → @主编 ←──────── 审稿No-Go → @写手
                                          ↑                              │
                                          │                              ↓
                                    @复盘官 ← @商务 ← @运营 ←── 审稿Go → @运营
                                        │
                                        └── 闭环 → @主编（新一轮）
```

### 人工确认点

| 节点 | 触发条件 | 谁确认 | 为什么 |
|------|---------|--------|--------|
| 董事长选题走查 | 主编出选题后 | 董事长（用户） | 方向错 = 全白干 |
| 设定走查2轮打回 | 编剧连续2轮打回设定 | 用户 | 需判断是设定问题还是选题问题 |

| 复盘闭环决策 | 复盘官出报告后 | @主编 | 继续/调整/砍掉 = 战略决策 |

## 反馈闭环协议（Feedback Loop Protocol）

在级联的关键交接点加入双向反馈循环，防止方向跑偏和需求漂移。走查（walkthrough）≠ 质疑（challenge），走查是快速确认"做得对不对"，质疑是深度拷问"做得好不好"。

### 核心规则

1. **走查门控** — 选题必须经董事长走查通过，设定必须经编剧走查通过，剧本必须经写手走查通过，才能级联到下游
2. **最多 2 轮回退** — 超过 2 轮上报用户
3. **走查记录** — 每次走查写入 `blackboard/walkthrough-{timestamp}.md`
4. **走查不替代质疑** — 质疑协议（challenge-protocol.md）照常运行，走查是额外的方向保险
5. **闭环优先级** — ① 选题走查 > ④ 设定走查 > ⑤ 剧本走查 > ② 审稿闸门 > ③ 复盘闭环

### 五条闭环

| 闭环 | 触发 | 走查者 | 被走查者 | 走查内容 | 最大轮数 |
|------|------|--------|----------|----------|----------|
| ① 选题走查 | 主编出选题 | 董事长(用户) | @主编 | 方向对不对？品类对不对？目标读者准不准？差异化够不够？ | 2 |
| ④ 设定走查 | 设定师出设定 | @编剧 | @设定师 | 世界观能撑50章？角色冲突够？风格可执行？ | 2 |
| ⑤ 剧本走查 | 编剧出剧本 | @写手 | @编剧 | 弧线每章有场景？爆点可写？节奏可用？ | 2 |
| ② 审稿闸门 | 写手出章节 | @审稿 | @写手 | P0/P1 问题清零 + 一致性 100% | 2（已有） |
| ③ 复盘闭环 | 复盘官出复盘 | @主编 | — | 主编决定继续/调整/砍掉 | —（已有） |

### 走查记录格式

```markdown
# 走查记录 walkthrough-{timestamp}
走查类型：选题走查 / 设定走查 / 剧本走查 / 审稿闸门 / 复盘闭环
走查者：[角色名]
被走查者：[角色名]
轮次：[1/2]

## 走查结论
通过 / 打回

## 走查要点
1. [要点1]：✅/❌ [说明]
2. [要点2]：✅/❌ [说明]
...

## 修改要求（打回时填写）
- [具体修改要求1]
- [具体修改要求2]
```

### 选题走查要点（董事长走查主编）

1. **方向一致性** — 选题是否符合公司战略方向和当前 OKR？
2. **品类差异化** — 是否有足够的差异化论证？不是"又一个XXX"
3. **目标读者** — 画像是否可验证？假设是否可证伪？
4. **可执行性** — 选题是否在现有资源下可实现？

### 设定走查要点（编剧走查设定师）

1. **故事可展开性** — 世界观规则和冲突源够不够撑50+章？角色关系有没有足够张力推进剧情？
2. **角色可写性** — 角色有没有致命缺陷和成长弧线？标志性语言是否可区分？
3. **风格可执行** — StyleGuide 对写手够不够具体？正面范例和反面禁忌可操作？
4. **情绪一致性** — 设定是否服务 EmotionMap 的核心情绪？不是设定师自嗨

### 剧本走查要点（写手走查编剧）

1. **场景可写** — 每章有明确场景和冲突？还是只有弧线框架没有落地细节？
2. **爆点可达** — 爆点位置有具体情节支撑？还是"这里该爆了"但没有写什么？
3. **节奏可执行** — 节奏图对写手是指导还是束缚？松紧交替是否自然？
4. **伏笔可埋** — LongTermPlan 的伏笔是否有明确的埋设方式和时机？

### 走查结果路由
