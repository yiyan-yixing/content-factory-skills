# 内容工厂 Agent 协作流程

> 8 层角色，1 条「战略 → 设定 → 剧本 → 生产 → 审稿 → 运营 → 商务 → 复盘」闭环 DAG。
> 支持：条件分支、质疑协议、共享记忆、DAG 编排。

---

## 核心闭环 DAG

不是线性流水线，而是支持条件分支的有向无环图。审稿是关键闸门，复盘驱动下一轮：

```
@主编 定义用户画像 + 情绪洞察 + 选题决策
  │
  └── @设定师 构建世界观 + 人物 + 关系 + 风格
        │
        └── @编剧 设计故事弧线 + 长线规划 + 爆点 + 连载节奏
              │
              └── @写手 场景设计 + 对话写作 + 章节生成 + 文风统一
                    │
                    ├── @审稿 结构审查 + 情绪强化 + 节奏优化 + 一致性检查
                    │     ├── 审稿通过 → @运营 ← 条件：publish_ready = true
                    │     └── 审稿不达标 → @写手 修改重写 ← 条件：P0/P1 问题未清零
                    │
                    └── @运营 读者转化 + 社群设计 + 留存 + 传播
                          │
                          └── @商务 会员设计 + 付费墙 + 产品化 + IP运营
                                │
                                └── @复盘官 反馈收集 + 内容复盘 + 技能优化 + 经验沉淀
                                      │
                                      └── 反馈闭环 → @主编（新一轮内容生产）
                                            ├── 继续 → 下一轮优化
                                            ├── 调整 → 修改选题/设定
                                            └── 砍掉 → @主编 切品类 → 回到 L0 重新定义
```

### 质疑节点（按 challenge-protocol.md）

```
@主编 出选题决策 ──→ @设定师 质疑设定可行性 ──→ 通过/修改/打回
                 ──→ @编剧 质疑故事可展开性 ──→ 通过/修改/打回

@设定师 出设定 ──→ @编剧 质疑设定是否足以支撑 50+ 章故事 ──→ 通过/修改/打回

@编剧 出剧本 ──→ @写手 质疑场景可执行性 ──→ 通过/修改/打回
              ──→ @审稿 质疑节奏和爆点是否合理 ──→ 通过/修改/打回

@写手 出章节 ──→ @审稿 质疑一致性和情绪强度 ──→ 通过/修改/打回

@商务 出付费墙位置 ──→ @运营 质疑是否打断读者体验 ──→ 通过/修改/打回
```

## 角色职责速查

| 角色 | 调用 | 做什么 | 不做什么 |
|------|------|--------|----------|
| **主编** | `@主编` | 定方向、画像、选题、情绪 | 不陷入写作细节 |
| **设定师** | `@设定师` | 世界观、人物、关系、风格 | 不自己写故事线 |
| **编剧** | `@编剧` | 故事弧线、伏笔、爆点、节奏 | 不亲自写每一章 |
| **写手** | `@写手` | 场景、对话、章节、文风 | 不改设定不改故事线 |
| **审稿** | `@审稿` | 结构、情绪、节奏、一致性 | 不跳过审查直接发布 |
| **运营** | `@运营` | 转化、社群、留存、传播 | 不刷量不追虚荣指标 |
| **商务** | `@商务` | 会员、付费墙、产品化、IP运营 | 不在平淡处卡付费墙 |
| **复盘官** | `@复盘官` | 反馈、复盘、优化、沉淀 | 不写流水账，要提取规则 |

## 快速出作品流程（1 周）

> 内容工厂铁律：没有可发布的章节，就没有读者。

```
Day 1: @主编 定画像+情绪+选题 + @设定师 出世界观+人物
       输入：目标平台 + 品类倾向
       输出：PersonaSheet + EmotionMap + TopicDecision + WorldBook + CharacterCard[]

Day 2: @设定师 出关系+风格 + @编剧 出故事弧线+爆点
       输入：L0+L1 产出
       输出：RelationMap + StyleGuide + StoryArc + HighlightList + RhythmChart

Day 3-4: @写手 批量产章节（5-10 章）
       输入：StoryArc + StyleGuide + RhythmChart
       输出：FinalChapter[]（5-10 章）

Day 5: @审稿 全链路审查 + @运营 出转化+传播方案
       输入：FinalChapter[] + 上游所有资产
       输出：publish_ready = true + ConversionPlan + SpreadPlan

Day 6-7: @商务 出付费墙+会员 + 发布上线 + 开始收集反馈
       输出：PaywallStrategy + SubscriptionPlan + 读者数据流
```

### 交接标准

| 交接 | 从 → 到 | 交接物 | 验收条件 |
|------|---------|--------|----------|
| 主编 → 设定师 | 选题 → 设定 | PersonaSheet + EmotionMap + TopicDecision | 画像有 3 个可验证假设，选题有差异化论证 |
| 设定师 → 编剧 | 设定 → 剧本 | WorldBook + CharacterCard[] + RelationMap + StyleGuide | 世界观有 3 条规则，角色有动机和缺陷 |
| 编剧 → 写手 | 剧本 → 生产 | StoryArc + LongTermPlan + HighlightList + RhythmChart | 故事有三幕结构，爆点间距 ≤ 5 章 |
| 写手 → 审稿 | 生产 → 审校 | FinalChapter[] | 每章 3000-5000 字，开头 200 字有钩子 |
| 审稿 → 运营 | 审校 → 发布 | publish_ready = true | P0/P1 问题清零，一致性 100% |
| 运营 → 商务 | 发布 → 变现 | ConversionPlan + CommunityPlan + SpreadPlan | 转化漏斗有量化目标 |
| 商务 → 复盘官 | 变现 → 复盘 | SubscriptionPlan + PaywallStrategy | 付费墙卡在情绪高点 |
| 复盘官 → 主编 | 复盘 → 新轮 | FeedbackDB + ReviewBook + KnowledgeBase | 每轮 ≥ 3 条可执行规则 |

## 一周一迭代节奏

```
周一（30min）          周三（15min）          周五（45min）
┌──────────┐         ┌──────────┐         ┌──────────────┐
│ 本期选题   │────────→│ 生产检查  │────────→│ 审稿+复盘    │
│ @主编 主导 │        │ @写手 反馈 │        │@审稿+@复盘官  │
│ @设定师 确认│       │ @编剧 协调 │        │@主编 校准    │
│ @编剧 可行  │       │           │        │              │
└──────────┘         └──────────┘         └──────────────┘
```

### 周一规划

- @主编 从 OKR 提取本期选题方向
- @设定师 确认设定就绪
- @编剧 确认故事弧线 + 本期爆点
- @写手 确认章节产出计划
- 质疑：@编剧 对选题做故事可展开性评审

### 周三检查

- @写手 反馈生产障碍（设定不够细/角色动机不清/节奏对不上）
- @编剧 协调调整
- 偏航 → 当天调整，不等到周五

### 周五复盘

- @审稿 出审稿报告（结构/情绪/节奏/一致性）
- @主编 判断内容假设是否成立
- @复盘官 提取本轮经验教训
- @主编 做战略校准（如需要）
- 定下期方向
- 写入 `.claude/blackboard/current-sprint.md`

## 紧急流程

### 严重一致性事故（人物/世界观前后矛盾已发布）

```
@审稿/读者 发现事故 → @写手 定位矛盾章节 → @审稿 评估影响范围
                     → @写手 修复 → @审稿 验证 → 重新发布
全程 < 2 小时止血
```

### 内容方向错误（读者大面积流失/差评）

```
@复盘官 报告数据异常 → @主编 判断是内容问题还是运营问题
                     → 内容问题 → @编剧 调整故事走向 + @写手 修改
                     → 方向问题 → @主编 切品类 → 回到 L0
不靠"再写几章看看"硬撑，数据说话
```

### 爆款机会窗口（某内容意外爆火/平台推流）

```
发现爆火信号 → @运营 加大传播投入 → @商务 提前设计付费墙和产品化
             → @主编 评估是否衍生新 IP → @设定师 快速扩展设定
抓住窗口期，48 小时内出商业方案
```
