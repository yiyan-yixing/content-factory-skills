# 质疑协议

> 内容工厂最大的风险是自嗨——写的人觉得好看，设定的人觉得够深。
> 质疑协议通过让不同 Agent 以对抗视角审查彼此的产出，弥补这个天然缺陷。
> 核心原则：**审稿先行，数据验证，不靠"我觉得"。**

---

## 触发条件

| 事件 | 必须质疑 | 质疑者 | 质疑维度 |
|------|----------|--------|----------|
| @主编 出选题决策 | ✅ 必须 | @设定师 | 设定可行性：这个选题能撑起 50+ 章世界观吗？ |
| | ✅ 必须 | @编剧 | 故事可展开性：这个选题能设计出足够的冲突和爆点吗？ |
| @设定师 出设定 | ✅ 必须 | @编剧 | 设定充分性：人物动机够强吗？关系有张力吗？能支撑长线？ |
| @编剧 出剧本 | ✅ 必须 | @写手 | 场景可执行性：场景能写出来吗？节奏图能落地吗？ |
| | ✅ 必须 | @审稿 | 节奏合理性：爆点间距合理吗？伏笔回收链完整吗？ |
| @写手 出章节 | ✅ 必须 | @审稿 | 一致性+情绪：人设前后一致吗？情绪强度达标吗？ |
| @商务 出付费墙位置 | ✅ 必须 | @运营 | 读者体验：付费墙会打断阅读体验吗？会不会导致弃书？ |
| @主编 做重大决策（切品类/换平台） | ✅ 必须 | @复盘官 | 数据支撑：有读者数据支撑这个决策吗？ |

## 质疑流程

```
1. 协调者（主对话）识别触发条件
2. 协调者派质疑 Agent 独立审查
3. 质疑 Agent：
   a. 阅读被质疑的产出（选题/设定/剧本/章节/付费墙/决策）
   b. 以对抗视角找出问题和风险
   c. 写入 .claude/blackboard/challenges.md
4. 被质疑者回应：
   a. 阅读 challenges.md 中的质疑
   b. 修改产出 OR 给出合理理由保留原方案
   c. 在 challenges.md 中填写回应
5. 协调者判断：
   - 通过 → 继续推进
   - 修改 → 修改后重新质疑
   - 打回 → 推翻重来
```

## 质疑者行为准则

- **质疑不是否定** — 目的是让产出更好，不是证明对方错了
- **找风险不找茬** — 关注"可能出什么问题"，不关注"措辞好不好"
- **给出替代方案** — 说"这个有问题"必须同时说"你还可以这样做"
- **标注严重度** — 阻断/重要/建议，不要所有问题都标成阻断
- **拿数据说话** — 质疑内容/设定必须引用画像/情绪/数据，不用直觉

## 质疑模板

质疑 Agent 的 system prompt 应包含：

```
你是质疑者。你的核心使命是以对抗视角审查 [对象]，找出可能的风险和问题。

审查维度：
1. [具体维度 1，如：设定可行性]
2. [具体维度 2，如：故事可展开性]

审查方式：
- 独立思考，不要顺着原作者的思路走
- 想想什么情况下这个内容/设定会失效
- 考虑边界和极端情况（长篇疲劳、读者流失、竞品同质化）

输出格式（写入 challenges.md）：
| 编号 | 日期 | 质疑者 | 对象 | 问题 | 严重度 | 回应 | 结论 |
```

## 内容工厂特有质疑维度

| 维度 | 关键问题 | 常见陷阱 |
|------|----------|----------|
| 设定可行性 | 这个选题能撑起 50+ 章吗？ | 选了太窄的题材，5 章就写完了 |
| 故事可展开性 | 冲突够吗？有足够转折点吗？ | 平铺直叙没有爆点 |
| 人物可投射性 | 读者能在角色身上看到自己吗？ | 角色太完美/太工具化，没有代入感 |
| 节奏合理性 | 爆点间距合理吗？有追更断裂点吗？ | 连续 5 章无进展，读者弃书 |
| 一致性 | 人设/世界观前后矛盾吗？ | 写了 30 章忘了前面的设定 |
| 情绪强度 | 关键场景能让读者"感受到"吗？ | 高潮场景只有叙述没有情绪 |
| 付费墙体验 | 付费墙在情绪高点吗？会弃书吗？ | 在平淡处卡墙，读者直接走 |
| 数据支撑 | 有完读率/跳出点数据支撑决策吗？ | 凭直觉改内容，不看数据 |

## 反模式

- ❌ 质疑者变成"橡皮图章"——走个形式就通过
- ❌ 被质疑者拒绝修改——"我觉得写得不错"
- ❌ 跳过审稿直接发布——"先发了看看反应"
- ❌ 所有质疑都标成"建议"——没有阻断级质疑说明质疑不够深入
- ❌ 质疑后不跟进——写了质疑但没人回应和判断
- ❌ 用直觉代替数据——"我觉得读者会喜欢"不是验证结论
